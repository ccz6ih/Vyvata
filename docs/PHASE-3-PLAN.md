## Overview

Phase 3 transforms Vyvata from a functional MVP into a revenue-generating B2B platform. This phase focuses on **practitioner retention and monetization** through enhanced patient management, professional deliverables, analytics, and billing.

### Business Goals
- Enable practitioners to manage 20+ patients efficiently
- Generate professional-grade patient deliverables (PDF reports)
- Provide cohort insights that demonstrate value
- Implement sustainable revenue model (Stripe subscriptions)
- Create viral referral loop for practitioner acquisition

### Success Metrics
- 3+ practitioners actively managing 10+ patients each
- PDF export used on 50%+ of patient interactions
- Cohort analytics viewed weekly by practitioners
- First paid practitioner subscription activated
- 2+ referral-driven practitioner signups

## Phase 3 Deliverables

### 1. Patient Management Enhancements (Week 1-2)

**What:** Rich patient data and workflow tools for practitioners.

**Technical Scope:**
- **Migration:** `patient_notes` table with FK to `patient_links`
- **Migration:** Add `status` column to `patient_links` (active | paused | archived)
- **API:** `PATCH /api/practitioner/patients/[id]/status` for transitions
- **API:** `POST /api/practitioner/patients/[id]/notes` for note creation
- **UI:** Notes section in [PatientDetailClient.tsx](src/app/practitioner/patients/[id]/PatientDetailClient.tsx)
- **UI:** Status badges and transition buttons
- **Feature:** CSV export of patient list from [DashboardClient.tsx](src/app/practitioner/dashboard/DashboardClient.tsx)

**Agent Assignment:** Supabase Guardian (migrations), Orchestrator (API + UI)

**Acceptance Criteria:**
- [ ] Practitioners can add timestamped notes to any patient
- [ ] Patient status transitions (active → paused → archived) reflected in dashboard counts
- [ ] CSV export includes all patient metadata, last protocol, created date
- [ ] Notes are private to the practitioner (RLS enforced)

### 2. PDF Export System (Week 2-3)

**What:** Professional PDF reports for practitioners to share with patients.

**Technical Scope:**
- **Dependency:** `react-pdf` or `@react-pdf/renderer` for PDF generation
- **API:** `GET /api/practitioner/patients/[id]/export-pdf` 
  - Generates PDF from protocol data
  - Includes practitioner branding (name, clinic)
  - Watermarks with protocol date and patient ID
- **UI:** "Export PDF" button in [PatientDetailClient.tsx](src/app/practitioner/patients/[id]/PatientDetailClient.tsx)
- **Design:** PDF template matching Vyvata brand (logo, colors, typography)

**Agent Assignment:** Orchestrator (research PDF libs + implementation)

**PDF Contents:**
- Header: Patient name, protocol date, practitioner details
- Goals & Stack: Formatted list of current supplements
- Rules Analysis: Interactions, synergies, redundancies, goal gaps
- Recommendations: Personalized synthesis (from LLM or fallback)
- Evidence: Top 3 evidence summaries for ingredients in stack
- Footer: "Generated by Vyvata • vyvata.com" + disclaimer

**Acceptance Criteria:**
- [ ] PDF export generates in <5 seconds for typical protocol
- [ ] PDF is print-ready (8.5x11, clean margins, readable at 100%)
- [ ] Practitioner name/clinic displayed prominently
- [ ] Export tracked in audit log
- [ ] PDF includes disclaimer: "For informational purposes only. Consult healthcare provider."

### 3. Cohort Analytics Dashboard (Week 3-4)

**What:** Aggregated insights across a practitioner's patient panel.

**Technical Scope:**
- **API:** `GET /api/practitioner/analytics/cohort`
  - Returns aggregated data for logged-in practitioner
  - Metrics: goal distribution, protocol distribution, average stack size, interaction frequency
- **UI:** New "Analytics" tab in [DashboardClient.tsx](src/app/practitioner/dashboard/DashboardClient.tsx)
- **Charts:** Goal distribution (bar chart), Top protocols (pie chart), Stack complexity histogram
- **Library:** Recharts or Chart.js for visualizations

**Agent Assignment:** Orchestrator (API + UI), Clinical Protocol Architect (interpretation guidance)

**Metrics to Display:**
1. **Goal Distribution** — What % of patients are optimizing for each of 7 goals?
2. **Protocol Distribution** — Which canonical protocols are most commonly matched?
3. **Stack Complexity** — Distribution of stack size (# ingredients) across patients
4. **Interaction Frequency** — What % of patients have critical/moderate interactions?
5. **Evidence Tier Breakdown** — What % of ingredients in cohort are strong/moderate/weak evidence?
6. **Trending Ingredients** — Top 10 ingredients across all patient stacks

**Acceptance Criteria:**
- [ ] Analytics load in <2 seconds for practitioners with 50+ patients
- [ ] Data updates in real-time when new patient added
- [ ] Charts are responsive and readable on mobile
- [ ] Export analytics as CSV or share as screenshot

### 4. Referral System (Week 4-5)

**What:** Viral loop for practitioner acquisition via unique share links.

**Technical Scope:**
- **Migration:** `referrals` table with referrer FK, referee FK, status (pending | converted), created_at
- **API:** `GET /api/practitioner/referral-link` — generates unique link with practitioner ID
- **API:** `POST /api/practitioner/register?ref=[code]` — tracks referral source
- **UI:** Referral section in dashboard showing:
  - Unique referral link (copy-to-clipboard)
  - Referral count (pending, converted)
  - Credits earned (if implementing credit system)
- **Feature:** Referral attribution tracked from signup → approval

**Agent Assignment:** Supabase Guardian (migration), Orchestrator (API + UI)

**Referral Mechanics (v1):**
- Each practitioner gets a unique referral code (e.g., `DEMO-REF-ABC123`)
- Referral link: `https://vyvata.com/practitioner/register?ref=ABC123`
- When new practitioner signs up with ref code, referrer gets notification
- Referral credit unlocks after referee reaches 5 active patients
- **Future:** Credits can unlock free months or advanced features

**Acceptance Criteria:**
- [ ] Referral link generates correctly for all practitioners
- [ ] Referrals tracked in database with pending/converted status
- [ ] Referrer receives email when referee is approved
- [ ] Dashboard shows referral count and status
- [ ] Referral code is URL-safe and unique

### 5. Stripe Billing Integration (Week 5-6)

**What:** Tiered subscription billing for practitioners based on patient volume.

**Technical Scope:**
- **Dependencies:** 
  - `stripe` npm package
  - Stripe webhook handler
- **Environment:** 
  - `STRIPE_SECRET_KEY`
  - `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`
  - `STRIPE_WEBHOOK_SECRET`
- **Migration:** Add `subscription_tier`, `subscription_status`, `stripe_customer_id` to `practitioners` table
- **API:** 
  - `POST /api/billing/create-checkout` — generates Stripe checkout session
  - `POST /api/billing/webhook` — handles Stripe events (payment success, subscription cancel)
  - `GET /api/billing/portal` — generates customer portal link for self-serve management
- **UI:** 
  - Billing section in dashboard showing current tier, usage, upgrade CTA
  - "Upgrade" button when approaching patient limit

**Agent Assignment:** Orchestrator (Stripe setup + webhooks), Supabase Guardian (migration)

**Subscription Tiers (v1):**

| Tier | Patient Limit | Price/Month | Features |
|------|--------------|-------------|----------|
| **Free** | 5 patients | $0 | Basic dashboard, CSV export |
| **Growth** | 25 patients | $49 | + PDF export, Analytics, Priority support |
| **Pro** | Unlimited | $149 | + Referral credits, White-label reports, API access |

**Stripe Workflow:**
1. Practitioner hits patient limit → "Upgrade" CTA in dashboard
2. Click → redirect to Stripe Checkout for Growth or Pro
3. Payment succeeds → webhook updates `subscription_tier` and `subscription_status`
4. Dashboard immediately reflects new limits and features
5. Practitioner can manage subscription via Stripe Customer Portal (cancel, update card)

**Acceptance Criteria:**
- [ ] Stripe test mode fully functional (use test cards)
- [ ] Webhook handler processes `checkout.session.completed` and `customer.subscription.updated`
- [ ] Patient limit enforced at API level (400 error when exceeded)
- [ ] Billing section shows current tier, patient usage (e.g., "12 / 25 patients"), days until renewal
- [ ] Customer portal link works for subscription management
- [ ] Stripe keys stored in environment variables (never committed)

## Risk Mitigation

### Technical Risks
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| PDF generation is slow (>10s) | Medium | High | Use streaming response, background job queue for large reports |
| Stripe webhook missed (network issue) | Low | Critical | Implement webhook retry logic, monitor Stripe dashboard for failed events |
| Cohort analytics query slow (N+1) | Medium | Medium | Add indexes on `patient_links.practitioner_id`, use aggregation queries |
| Referral code collision | Low | Low | Use UUID-based codes, check uniqueness before insert |

### Business Risks
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Pricing too low (undervalued) | Medium | High | Start conservative, survey first 10 practitioners for willingness to pay |
| Practitioners churn after free tier | High | Medium | Build referral credits to incentivize growth, offer annual discount |
| Stripe fees eat margins | Medium | Medium | Price tiers to absorb 2.9% + $0.30 fee |
